## What changed (design)

### 1. Defensive MAX_MAKERS cap (in code, not prompt-only)
The picker's returned task list is capped with `.slice(0, CONFIG.maxMakers)` in code in BOTH the
dry-run path and the live drain loop, so an over-eager picker can never spawn more than
`MAX_MAKERS` concurrent makers regardless of what it returns. Carry-forward from the O5 review.

### 2. Drain-then-stop with PER_RUN_TASK_CAP
The pickâ†’developâ†’integrate cycle is wrapped in an OUTER `while (true)` loop. Each iteration runs
one picker round (up to `MAX_MAKERS` after the defensive slice) via `runRound()`. The loop
repeats until EITHER the picker returns no eligible work (`stopReason = "drained"`) OR the
cumulative number of ATTEMPTED tasks reaches `CONFIG.perRunTaskCap` (`"cap-reached"`). The cap is
enforced two ways: the top-of-loop guard `if (attempted >= perRunTaskCap) break`, and a per-round
truncation `eligible.slice(0, perRunTaskCap - attempted)` so the final round runs FEWER than
`MAX_MAKERS` if needed to land exactly on the cap (e.g. cap 10, maxMakers 3 â†’ rounds attempt
3,3,3,1). On stop it emits a `log()` line: `drain-then-stop: processed N task(s) across R
completed round(s); stopped â€” PER_RUN_TASK_CAP=â€¦ reached.` / `â€¦ no eligible work remaining
(drained).` Each round's picker cannot re-pick already-handled tasks: `runRound()` fully resolves
every task to Done (leaves `Todo`) or Blocked/Needs-human (carries `agent:needs-human`, which the
picker excludes) before the next round's picker runs â€” so no task is attempted twice. If nothing
is ever eligible (first round empty) the prior `{status:"NO_ELIGIBLE_WORK"}` contract is preserved.

### 3. Crash / RETRY handling (a dead agent is never a verdict)
A new `crashed(item, why)` marker (`state:"CRASHED"`) is returned whenever an `agent()` hook
returns `null` (the agent died) â€” from the initial maker (`developTask`), the reviewer or the
fix-maker (`runReviewLoop`), or the integration-fix maker (`reintegrateFix`). Critically, a
**null reviewer verdict is NOT bounced** (the old code's `review && verdict==="PASS"` else-bounce
path silently treated a dead reviewer as CHANGES-REQUESTED); it now surfaces as a crash. A new
`withCrashRetry(pipelineThunk, phaseLabel)` wraps each per-task pipeline (develop and
integration-fix), retrying up to `CONFIG.retry` times on a `CRASHED` result. When RETRY is
exhausted it escalates the task to `agent:needs-human` (Blocked) with reason `agent-crash` and
returns a `blocked()` result so the caller CONTINUES with the other tasks â€” the run never hangs
and one dead agent never fails the others. A crash does NOT increment the bounce counter (it is
not a review bounce). Non-crash failures keep their existing terminal reasons
(`NOT_DEVELOPABLE`, `BLOCKED`, `maker-failed-on-changes`, `review-bounce-limit`,
`maker-failed-on-integration-fix`, `integration-bounce-limit`).

### 4. Per-agent timeout â€” N/A (runtime limitation)
A true per-agent wall-clock timeout is **N/A â€” runtime limitation**: the Workflow `agent()` hook
exposes no timeout option, so it cannot be enforced in-script. This is documented with an inline
comment on `CONFIG.agentTimeoutMinutes` (recorded for documentation only; `AGENT_TIMEOUT`=20m from
`config.md`, never read for control flow) and on `withCrashRetry()`. The resilience mechanism is
the null-return crash handling above: a hung agent the runtime eventually abandons surfaces as a
`null` return, which is retried up to `RETRY` and then escalated. No fake timeout was added.

### 5. Config, not hard-coded
`CONFIG.perRunTaskCap` (default 10, `args.perRunTaskCap`), `CONFIG.retry` (default 1,
`args.retry`), and `CONFIG.agentTimeoutMinutes` (default 20, doc-only) join the existing
`maxMakers`/`bounceLimit`. A comment block over `CONFIG` names `docs/orchestrator/config.md` as the
source of the default values (MAX_MAKERS=3, BOUNCE_LIMIT=2, PER_RUN_TASK_CAP=10, RETRY=1,
AGENT_TIMEOUT=20m). No limit is hard-coded at a call site.

### Preserved v2 guarantees
`meta` is still a pure string literal (phases Pick/Develop/Integrate, repeated per round). No
`Date.now`/`Math.random`/`new Date`/filesystem/`process`/`require`/`import` in the script body â€”
all git + Linear work happens inside dispatched agent prompts. LF line endings preserved
(`.gitattributes eol=lf`; verified 0 CR bytes). `dryRun` still dispatches ONLY the picker and
returns with no side effects (now also reporting `perRunTaskCap` and `retry`). Return value is
the finished-summaries array, one entry per attempted task.

## Decisions / assumptions
- **A crash resets the per-task pipeline, not the bounce counter.** `withCrashRetry` re-runs the
  whole `developTask`/`reintegrateFix` thunk on a `CRASHED` result â€” consistent with Â§4's
  "crash/timeout â†’ back to Todo (fresh attempt)". `RETRY=1` therefore means initial attempt + 1
  retry = 2 dispatches before escalation. Crashes never touch `item.bounces` (which governs review
  bounces only).
- **`attempted` counts tasks the run took ownership of** (set to `In Progress`), matching "max
  tasks attempted in one run". A task that later crashes/blocks still counts as attempted.
- **`agentTimeoutMinutes` is doc-only.** Kept in CONFIG so the value has a single recorded home and
  is args-overridable, but it is explicitly commented as never read for control flow (the hook has
  no timeout knob). This is the honest representation of the N/A limitation, not a stub timer.
- **`NO_ELIGIBLE_WORK` contract preserved** for a first-round-empty run (returns the object, not an
  empty array) so callers/`/loop` see the same shape as v1/v2.
- Retained from v2: `agentType:'general-purpose'` with inlined contracts (the runtime does not load
  `.claude/agents/*.md` as agentType), single per-task bounce counter, non-blocking rework between
  merge passes, `parallel(arrayOfThunks)` convention with a `Promise.all` fallback in `fanOut`.

## Known risks / follow-ups
- **`parallel()` calling convention unverified against the live runtime** (inherited from O5): the
  `Promise.all` fallback keeps the script correct if `parallel` is absent, but if the runtime
  injects a `parallel` expecting already-started promises the wrapper passes thunks. Low impact.
- **Crash detection depends on the runtime returning `null` for a dead/abandoned agent.** If the
  runtime instead throws or hangs indefinitely (never returning), the in-script retry cannot fire â€”
  the true wall-clock timeout that would cover that case is the N/A limitation above. First live run
  should confirm the runtime surfaces agent death as a `null` return.
- **Cross-round re-pick safety depends on Linear state propagation** â€” a task must be off `Todo`
  (Done) or labelled `agent:needs-human` before the next round's picker query. This holds because
  `runRound` awaits every task to a terminal state before returning; confirm on the first live
  multi-round run that Linear writes are visible to the immediately-following picker.
- Live behavior (real Linear transitions, real concurrent merges, real agent crashes) is unverified
  by design â€” this is build-only. First live validation is a supervised dry-run then a seeded run.
