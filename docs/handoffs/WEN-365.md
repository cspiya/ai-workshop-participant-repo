## What changed (the audit trail)

### 1. Done-evidence comment (integrator prompt)
`integratorPrompt()` now instructs the integrator to POST A REQUIRED Linear comment
(explicitly via the Linear MCP create/save-comment tool, loaded with ToolSearch) on
the Done transition, containing all four evidence fields, each on its own line:
- **Merge commit SHA on main** — the exact merge commit created in step 2, now on
  `origin/main`.
- **Gate exit codes** — the four codes captured re-running the gate on merged main,
  written as `typecheck=<n> lint=<n> test=<n> build=<n>` (all 0 for a Done issue).
- **Handoff link** — `docs/handoffs/<ISSUE-ID>.md` (the maker's evidence trail).
- **Production deploy URL** — `https://ai-workshop-participant-repo-sand.vercel.app`,
  with an explicit note that the git-linked Vercel auto-deploys on push to main so the
  specific deploy for that commit MAY STILL BE BUILDING at comment time.
The step is marked REQUIRED ("the Done transition is not complete without it"), so the
integrator cannot silently skip it.

### 2. Blocked / needs-human comment (central `escalate()` helper)
Every terminal escalation path routes through the single `escalate(item, note)` helper
(verified: NOT_DEVELOPABLE / BLOCKED in `developTask`, review bounce-limit and
maker-cannot-comply in `runReviewLoop`, crash-after-RETRY in `withCrashRetry`,
integration conflict/red-gate exceeding the bounce limit in `runRound`, and
maker-failed-on-integration-fix in `reintegrateFix`). `escalate()` now BOTH:
- sets label `agent:needs-human` (via `setLinearState`, unchanged), AND
- posts a Linear comment stating the EXACT blocking reason / the reviewer's
  Critical+Serious findings (or the crash/conflict reason) passed in as `note`, and
  NAMES the human owner **Csaba Piya (product owner)**, noting that per
  docs/agentic-operating-model.md §9 only a human resolves a Blocked/Needs-human task.
The label alone is no longer sufficient — the comment is always posted alongside it.

Two pure-literal constants were added (`HUMAN_OWNER`, `PROD_DEPLOY_URL`) — no
`Date`/`Math.random`/filesystem access introduced.

## Decisions / assumptions
- Centralised the owner-naming + comment in `escalate()` rather than at each call site,
  so every escalation path (present and future) posts the comment — DRY and no missed
  path. (Owner naming value `Csaba Piya (product owner)` per task brief.)
- The orchestrator body is a Workflow script and is not compiled by the repo gate
  (`tsc` covers `src`, eslint/vitest/next build do not include `.claude/workflows`).
  Syntax validated separately with `node --check` on a wrapped copy → SYNTAX_OK.

## Known risks / follow-ups
- The specific Vercel deploy URL for a given merge commit is not resolved
  programmatically; the comment always includes the stable production URL plus a
  "may still be building" note. If the Vercel MCP is later wired in, the integrator
  could resolve the exact per-deploy URL — out of scope for O8.
